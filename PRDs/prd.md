这份产品需求文档（PRD）基于 **PostGIS/pgRouting（数据与计算层） + Rust/Tokio（服务层） + Redis（缓存层）** 技术架构进行编制，旨在明确业务需求、系统边界与技术实现指标。

# 公园内部道路导航系统产品需求文档 (PRD)

## 1. 文档概述
### 1.1 背景与痛点
公园（如大型景区、主题乐园）内部路网不同于市政道路，具有人行步道多、车辆禁行区多、景点与公共设施分布密集的特点。在节假日等游客高峰期，系统面临高并发的路径规划请求。
### 1.2 建设目标
构建一个轻量、稳定、低延迟的公园内部道路导航系统，在保障数据准确性的同时，能够平稳应对高并发流量，提升游客的游园体验。

---

## 2. 核心业务场景与用户用例
*   **场景 A：精准导航**
    游客在公园内任意位置，输入目的地（如特定景点、洗手间、出口），系统规划出步行或电瓶车路线。
*   **场景 B：高峰期并发**
    节假日离园高峰期，大量游客同时发起“当前位置至各出口”的导航请求。
*   **场景 C：推荐游览路线**
    系统提供公园预设的经典游览路线（如“一日游”、“亲子游”），用户可直接一键导航。

---

## 3. 功能需求 (FR)

### 3.1 路径规划引擎 (Core API)
*   **输入参数**：起点坐标 $(Lon_1, Lat_1)$、终点坐标 $(Lon_2, Lat_2)$、出行方式（步行/电瓶车）。
*   **输出参数**：导航路径几何点集（GeoJSON LineString）、总距离（米）、预计耗时（分钟）。AR 导航另由 `POST /api/v1/navigate` 提供高密度折线与转向步骤。
*   **路由逻辑**：
    1.  **近邻匹配（Snap）**：将输入坐标投影至最近的路网边（Edge）。
    2.  **路径计算**：调用 pgRouting 算法（如 Dijkstra 或 A*）计算最优路径。
    3.  **结果返回**：格式化为标准 GeoJSON 格式供前端渲染。

### 3.2 热门路线与静态缓存管理
*   **经典路线管理**：支持管理员在后台预设多条推荐路线。
*   **高频起终点缓存**：系统需自动识别并缓存“景点 A ➔ 景点 B”、“当前位置 ➔ 最近出口”等高频查询结果。

### 3.3 路网拓扑维护（后台功能）
*   **路网导入**：支持导入 Shapefile 或 GeoJSON 格式的公园路网数据。
*   **拓扑构建**：系统导入新路网后，需自动或手动触发 `pgr_createTopology` 重建拓扑关系，并进行连通性自检。

## 4. 技术架构与数据流设计

为了兼顾开发效率、维护成本与高并发性能，系统采用以下三层架构：

```
+-------------------------------------------------------------+
|                      前端/客户端 (小程序/H5)                 |
+-------------------------------------------------------------+
                               |  HTTP / gRPC
                               v
+-------------------------------------------------------------+
|             服务层：Rust (Actix-web / Axum) + Tokio          |
|  - 接收高并发请求                                            |
|  - 鉴权、限流、参数校验                                       |
|  - Redis 缓存命中判断                                        |
+-------------------------------------------------------------+
         |                                           |
         | (1) 缓存命中                               | (2) 缓存未命中
         v                                           v
+------------------------+                 +------------------------+
|   缓存层：Redis         |                 | 数据/计算层：PostgreSQL |
|  - 存储高频 O-D 路径    |                 |  - PostGIS (空间检索)  |
|  - 热门路线静态数据     |                 |  - pgRouting (算法计算) |
+------------------------+                 +------------------------+
         |                                           |
         +<------------ 将新计算路径回写 ------------+
```

### 4.1 数据流详细说明
1.  **请求分流**：Rust 服务端接收到起点 $A$ 和终点 $B$ 的导航请求。
2.  **缓存检索**：
    *   若 $A, B$ 为特定 POI（如“南门”到“大熊猫馆”），或者属于已缓存的网格区域，Rust 服务端直接从 Redis 读取路径数据并返回。
    *   若未命中缓存，则进入数据库查询流程。
3.  **实时计算与优化**：
    *   **空间过滤**：Rust 服务向 PostgreSQL 发起 SQL 查询，利用 PostGIS 的 `ST_Buffer` 或 `ST_Envelope` 限定计算路网的边界框（Bounding Box），避免全图计算。
    *   **拓扑寻路**：在限定的路网子集中，调用 `pgr_dijkstra` 计算路径。
    *   **回写缓存**：计算结果返回给游客的同时，异步写入 Redis，供后续邻近请求复用。

## 5. 非功能性需求 (NFR)

### 5.1 性能与响应指标
*   **高频/缓存请求响应时间**：平均响应时间 $\le 50\text{ms}$（Redis 读取）。
*   **实时路径计算响应时间**：在路网数据限制在局部区域的前提下，单次 `pgRouting` 计算响应时间 $\le 300\text{ms}$。
*   **并发承载能力**：
    *   利用 Rust 的异步 IO 特性，配合 Redis 缓存，系统设计承载吞吐量需达到 $\ge 2000\text{ QPS}$。

### 5.2 数据精度与可用性
*   **路网连通性**：建图时必须保证无悬挂点、孤立线，连通率需达到 $99.9\%$。
*   **降级方案**：若数据库发生偶发性延迟，系统优先返回 Redis 中存储的“景区推荐路线”或“邻近主干道导航”，避免服务中断。

## 6. 关键数据表结构设计（草案）

### 6.1 路网边表 (`park_ways`)
用于 pgRouting 计算的核心数据表：

| 字段名 | 类型 | 说明 | 约束/索引 |
| :--- | :--- | :--- | :--- |
| `id` | SERIAL | 主键 | PRIMARY KEY |
| `source` | INTEGER | 起点节点 ID | INDEX (pgr 拓扑使用) |
| `target` | INTEGER | 终点节点 ID | INDEX (pgr 拓扑使用) |
| `cost` | DOUBLE PRECISION | 正向通行权重（距离或时间） | - |
| `reverse_cost` | DOUBLE PRECISION | 反向通行权重 | - |
| `the_geom` | GEOMETRY(LineString, 4326) | 空间几何线段 | GiST 空间索引 |

### 6.2 缓存 Key 设计 (`Redis`)
*   **Key 格式**：`nav:path:{start_poi_id}:{end_poi_id}:{travel_mode}`
*   **Value 格式**：压缩后的 GeoJSON 字符串或 Protocol Buffers 字节流。
*   **有效期**：对于固定 POI 路由，设置较长有效期（如 24 小时）；对于动态坐标寻路缓存，设置较短有效期（如 10 分钟）。
